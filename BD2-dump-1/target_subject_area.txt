Общая архитектура

Целевая модель данных построена на сочетании трёх подходов к моделированию и содержит 14 таблиц. Предметная область — человек как покупатель маркетплейса техники.

---
Star Schema — центр и лучи

Таблица persons является центром «звезды». Она хранит основные анкетные данные персоны и содержит внешние ключи на справочные таблицы-«лучи»:

┌─────────────────────┬──────────────────────────────────────────────────────┐
│ Таблица-справочник  │                      Что хранит                      │
├─────────────────────┼──────────────────────────────────────────────────────┤
│ dim_gender          │ Пол: M / F / OTHER                                   │
├─────────────────────┼──────────────────────────────────────────────────────┤
│ dim_marital_stat
│ dim_contact_type    │ Тип контакта: телефон, email, Telegram и др.         │
├─────────────────────┼──────────────────────────────────────────────────────┤
│ dim_address_type    │ Тип адреса: доставка, регистрация, фактический       │
├─────────────────────┼──────────────────────────────────────────────────────┤
│ dim_document_type   │ Тип документа: паспорт РФ, загранпаспорт, ВУ и др.   │
├─────────────────────┼──────────────────────────────────────────────────────┤
│ dim_consent_type    │ Тип согласия: обработка ПД, маркетинг, партнёры      │
└─────────────────────┴──────────────────────────────────────────────────────┘

Центральная таблица persons содержит:
- Разбитое ФИО (last_name, first_name, middle_name) + исходная строка full_name_raw — на случай если источник передал ФИО нечитаемым образом
- Нормализованная дата рождения birth_date (тип DATE) + birth_date_raw (VARCHAR) — для хранения исходного грязного значения вроде «5 января 1990 года»
- ИНН и СНИЛС — в двух вариантах: нормализованный и исходный (inn_raw, snils_raw)
- Служебные поля: created_at, updated_at, is_deleted, deleted_at

Все первичные и внешние ключи во всех таблицах — тип UUID.

---
Snowflake Schema — трёхуровневая иерархия адреса

Адресное пространство организовано в виде снежинки с тремя уровнями:

dim_country  (страна)
    └── dim_region  (регион / субъект РФ)
            └── diструктурированных полей, она содержит address_raw — исходную строку адреса из источника без разбора. Один человек может иметь несколько адресов разных типов (доставка, регистрация, фактический).

---
EAV — гибкие атрибуты покупателя

Для хранения признаков, специфичных для маркетплейса техники и необязательных для каждого покупателя, применяется подход Entity-Attribute-Value:

person_attribute_types — каталог типов атрибутов. Каждый тип имеет код, тип данных (text, number, boolean, date, json) и категорию (buyer_preference, marketing, technical).

Примеры атрибутов в каталоге:

┌────────────────────────┬─────────────────────────────┬────────────┐
│          Код           │          Описание           │ Тип данных │
├────────────────────────┼─────────────────────────────┼────────────┤
│ fav_tech_category      │ Любимая категория техники   │ text       │
├────────────────────────┼─────────────────────────────┼────────────┤
│ preferred_brand        │ Предпочитаемый бренд        │ text       │
├────────────────────────┼─────────────────────────────┼────────────┤
│ interested_installment │ Интерес к рассрочке         │ boolean    │
├────────────────────────┼─────────────────────────────┼────────────┤
│ avg_budget_rub         │ Средний бюджет покупки      │ number     │
├────────────────────────┼─────────────────────────────┼────────────┤
│ has_smart_home         │ Есть устройства умного дома │ boolean    │
├────────────────────────┼─────────────────────────────┼────────────┤
│ preferred_ecosystem    │ Предпочитаемая экосистема   │ text       │
├────────────────────────┼─────────────────────────────┼────────────┤
│ risk_score             │ Скоринговый балл            │ number     │
└────────────────────────┴─────────────────────────────┴────────────┘

person_attributes — таблица значений. Содержит пять колонок-значений под разные типы данных (value_text, value_number, value_boolean, value_date, value_json), поля версионирования (valid_from, valid_to) и ссылки на персону и тип атрибута.

---
Таблица логов миграции

migration_log фиксирует каждую операцию переноса данных:

- migration_run_id — идентификатор запуска, группирует записи одной сессии
- source_system, source_record_id — откуда пришла запись
- target_person_id — созданная персона (NULL при ошибке)
- status — success / warning / error / skipped
- stage — этап: extract, transform, load, validate
- error_code, error_message — детали ошибки
- warning_messages — массив предупреждений (нераспознанная дата, невалидный СНИЛС и т.п.)
- source_data (JSONB) — снимок исходной записи для аудита и повторной обработки

---
Единственная точка создания персоны

Создание новой записи в целевой БД производится исключительно через вызов функции create_person(). Функция атомарно создаёт записи в таблицах persons и person_accounts, добавляет первичные контакты (телефон, email), нормализует дату рождения, СНИЛС, ИНН и телефон, и фиксирует результат в migration_log. Возвращает person_id созданной персоны.
