# Анализ миграции данных партнера

## Совместимость предметных областей

Предметные области хорошо совпадают: обе базы описывают человека как покупателя маркетплейса техники.

Совпадающие смысловые группы:

- ФИО и дата рождения;
- пол;
- аккаунт покупателя;
- контакты;
- ИНН и СНИЛС;
- адреса;
- документы;
- согласия;
- дополнительные покупательские признаки.

Основное различие не в предметной области, а в физической структуре:

- у партнера данные хранятся проще и грязнее, часть значений лежит одной строкой;
- у нас модель нормализована, есть справочники, snowflake для адресов и business-EAV.

Это соответствует смыслу задания: данные мапятся, но требуют преобразований.

## Источник партнера

Фактическая source-БД партнера из `BD2-dump-1/05_source_db_export.sql` содержит 5 таблиц:

- `customers` - основные данные покупателя;
- `cust_addresses` - адреса свободной строкой;
- `cust_docs` - документы одной строкой;
- `cust_consents` - согласия с разными форматами значений;
- `cust_extra` - дополнительные EAV-подобные признаки.

Количество исходных строк:

```text
customers: 25
cust_addresses: 20
cust_docs: 15
cust_consents: 25
cust_extra: 25
```

## Основной маппинг

| Источник партнера | Целевая модель | Преобразование |
|---|---|---|
| `customers.fullname` | `person_profile.last_name`, `first_name`, `middle_name` | Разбиение ФИО по пробелам, приведение регистра |
| `customers.birth_dt` | `person_profile.birth_date`, `birth_date_raw` | Парсинг известных форматов даты, исходная строка сохраняется |
| `customers.sex` | `person_profile.gender_id` | Маппинг значений `М`, `Ж`, `male`, `female`, полных слов |
| `customers.login`, `status` | `user_account` | Статус приводится к `ACTIVE` / `BLOCKED` |
| `customers.email`, `phone`, `phone2` | `user_contact` | Разделение нескольких email, нормализация телефона до цифр |
| `customers.inn`, `snils` | `person_identifier` | Удаление пробелов, дефисов и прочих нецифровых символов |
| `cust_addresses` | `user_address` + address snowflake | Грубое выделение города, региона, улицы, дома, квартиры; raw-адрес сохраняется |
| `cust_docs` | `user_verification_document` | Тип документа мапится, серия/номер/дата извлекаются регулярными выражениями |
| `cust_consents` | `user_consent` | `да`, `1`, `yes`, `true`, `+` -> true; `нет`, `0`, `no`, `false`, `-` -> false |
| `cust_extra` | `user_attribute_value` | Известные признаки переносятся в business-EAV |
| `customers.marital` | `map.migration_unmapped_attribute` | В нашей модели нет семейного положения |

## Результат прогона

Миграция выполнялась скриптом:

```text
sql/run_partner_migration.sql
```

Результат:

```text
source_customers: 25
migrated_links: 25
migration_log_rows: 25
target_persons_total: 37
unmapped_attributes: 25
```

`target_persons_total = 37`, потому что в нашей БД уже было 25 собственных seed-покупателей, а часть записей партнера была сопоставлена с уже существующими людьми по логину, контактам, идентификаторам или ФИО с датой рождения. Новыми людьми стали только те записи партнера, для которых совпадение не найдено.

Статусы логов:

```text
success: 13
warning: 12
error: 0
```

То есть все 25 строк партнера успешно перенесены, но 12 строк потребовали предупреждений.

## Проверка на дублирование

Для защиты от дублей используются два уровня проверки:

- `map.migration_person_link` хранит связь `source_system + source_record_id -> target_person_id`;
- `find_marketplace_customer(...)` сверяет новую запись с нашей бизнес-БД по логину, идентификаторам, контактам и ФИО с датой рождения.

Повторный запуск миграции по тем же данным не создает новых людей. Уже перенесенные source-записи получают статус `skipped`.

Результат повторного запуска:

```text
skipped: 25
target_persons_total: 37
```

Дополнительные проверки дублей после повторного запуска:

```text
duplicate_logins: 0
duplicate_profile_natural_key: 0
duplicate_identifiers: 0
duplicate_contacts_per_person: 0
duplicate_addresses_per_person: 0
duplicate_documents_per_person: 0
```

## Демонстрационный прогон с ошибками

Основной прогон миграции завершился без ошибок. Чтобы выполнить требование отчета о демонстрации `error`-записей в `migration_log`, был сделан отдельный демонстрационный сценарий:

```text
sql/run_partner_migration_error_demo.sql
```

Сценарий сначала выполняет обычную миграцию, затем добавляет в `map.customers` две намеренно некорректные строки:

- `cust_id = 9001` - пустое ФИО;
- `cust_id = 9002` - отсутствует `login`.

После повторного вызова `map.migrate_partner_customers()` эти строки не попадают в целевую модель, а фиксируются в `map.migration_log` со статусом `error`.

Результат демонстрационного прогона:

```text
success: 13
warning: 12
skipped: 25
error: 2
```

Ошибки:

```text
9001: last_name and first_name are required
9002: login is required
```

Метрики после демонстрационного error-прогона:

```text
source_customers: 27
migrated_links: 25
target_persons_total: 37
migration_log_rows: 52
unmapped_attributes: 25
```

`target_persons_total` не увеличился, потому что ошибочные строки были отклонены процедурой создания человека и остались только в логах.

## Предупреждения

Примеры предупреждений:

- ФИО содержит инициалы: `Козлов А.Е.`, `Морозов Д.А.`, `Белова К.`;
- дата рождения не распознана: `ноябрь 1979`, `н/д`;
- адрес отсутствует или пустой;
- подозрительный email: `marinakrylova.mail.ru`.

Такие строки не блокируют миграцию, потому что исходное значение сохраняется в raw-полях или логах.

## Несопоставленные данные

В `map.migration_unmapped_attribute` попали:

- `customers.marital` - семейное положение, так как в модели покупателя маркетплейса техники оно не хранится;
- `cust_extra.Sony = да` - некорректный EAV-ключ партнера, где ключ и значение выглядят перепутанными.

## Вывод

Миграция между нашими БД успешна и осмысленна:

- предметные области совпадают;
- данные партнера хорошо ложатся на нашу модель;
- несовпадения структуры требуют реального маппинга и очистки;
- грязные данные не теряются, а сохраняются через raw-поля, warning-логи и таблицу несопоставленных атрибутов.
